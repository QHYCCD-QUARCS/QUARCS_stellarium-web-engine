# AI-Control

`AI-Control/` 是一个不侵入 `tests/` 的新型 E2E/CLI 业务控制层草案。

目标：
- 在不修改现有 `tests/e2e/flows` 的前提下，建立新的多层分工结构。
- 把“主菜单、子菜单、连接、分配、拍摄、保存”等动作拆成可独立调用的节点。
- 让高层业务 flow 可以由这些细粒度步骤自由组合成流程网。

目录说明：
- `core/`：流程上下文、步骤类型、运行器与 registry 合并。
- `shared/`：交互原语、导航、guard、等待。
- `atomic/`：通用 UI 原子步骤。
- `menu/`：主菜单、子菜单、确认弹窗、设置入口。
- `device/`：设备侧栏、连接、分配、拍摄、QHY 别名。
- `scenario/`：面向 CLI 或 MCP 的高层业务 flow builder。

约束：
- 所有交互必须走真实链路，不使用 `force`。
- 每个关键步骤都要求具备前置检查、执行动作、后置确认。
- `AI-Control/` 仅提供新实现草案，不替换现有 `tests/` 目录逻辑。

## 制作规则

`AI-Control` 的步骤设计、Flow 组织与 CLI 对外接口，统一遵循以下规则。

### 1. 设计原则

- 业务动作优先：不要围绕单个页面控件拼接脚本，而要围绕“连接设备、进入配置、分配设备、开始拍摄、保存结果”等可验证业务动作设计稳定接口。
- 先判断前置要求：执行前必须检查页面模块、数据加载、目标对象、当前状态、依赖步骤等是否满足；不满足时要明确返回阻塞原因，不能强推执行。
- 前置步骤必须可复用：打开菜单、进入页面、加载列表、选择目标等关键准备动作要拆成独立步骤，不能隐藏在主流程内部。
- 先做可操作性检查：交互前要确认目标可见、可用、未被遮挡、已渲染完成，再执行点击、输入、选择等动作。
- 后置确认必做：不能把“点击成功”当成完成标准，必须验证页面状态、文案、标记、对象状态、数据刷新或提示信息是否真实变化。
- 状态变化可追踪：关键动作需要体现“执行前状态 -> 触发动作 -> 执行后状态 -> 是否符合预期”，并在日志、断言或返回结果中保留依据。

### 2. 交互实现要求

- 不凭猜测模拟控件：优先阅读源码，确认真实触发方式，例如 `@click`、`v-model`、自定义事件或组件封装行为。
- 若控件由 Vuetify 等组件库封装，要确认 `data-testid` 所在节点是否真能点击；若绑定在隐藏 input 上，应点击其可见父容器。
- 若交互依赖展开态、激活态、hover、焦点或异步渲染完成，必须先满足这些前提，再执行后续动作。
- 对菜单、子菜单、弹窗、下拉、标签页等层级控件，必须按真实业务顺序逐级进入，不得跳过链路直接打目标节点。
- 严禁使用 `force: true` 或其他绕过真实交互约束的方式。

### 3. 元素定位规则

- 正式流程统一以全局唯一的 `data-testid` 为主定位方式。
- 若控件缺少 `data-testid`，应先回到源码补齐，再纳入 `AI-Control`、Flow 或 CLI。
- 命名要有明确业务语义，作用域清晰，并与现有前缀体系保持一致，如 `pa-`、`cp-`、`gui-`、`ui-`。
- 不允许把文本内容、DOM 层级、样式类名或随机结构作为正式依赖，它们只能用于临时排查。
- 新增或修改 testid 时，应同时参考并按需更新以下报告或索引：
  - `apps/web-frontend/docs/testid-validation-report.md`
  - `apps/web-frontend/docs/testid-scan-report.md`
  - `E2E_TEST_IDS_INDEX.json`

### 4. Flow / CLI 结构

- 每个业务功能都应拆成五段：前置检查、前置步骤、执行动作、后置确认、状态校验。
- 前置检查、前置步骤、执行动作、后置确认应尽量保持为独立函数、独立 action 或独立 flow step，避免堆进一个大函数。
- 若支持重试，必须基于明确状态判断后再重试，不能盲目重复点击。
- 若存在失败分支、异常提示、权限限制或空数据场景，流程里必须显式处理，不能只覆盖理想路径。

### 5. 实施细则

- 点击：优先使用等价于 `scrollIntoViewIfNeeded -> visible/assert enabled -> click` 的真实交互链路。
- 输入：先确保目标可见且可交互，再执行 `fill` / `type`。
- 下拉与选择：通过 testid 找到可见触发区域或选项后再点击，不直接对隐藏 input 做强制操作。
- 菜单与弹层：先确认主入口已展开，再定位子项；若存在动画或挂载延迟，需等待进入可交互状态。
- 异步流程：提交、切换、加载、刷新后，必须等待明确完成信号，例如 loading 消失、按钮恢复、列表更新、状态标签变化、提示出现。

### 6. 断言、日志与职责边界

- 断言优先围绕业务结果，而不是“页面点到了”。例如是否真正连接成功、保存成功、切换到目标模式或进入目标页面。
- 关键流程建议同时校验页面反馈、内部状态与业务结果，避免出现 UI 有动作但业务未生效的假阳性。
- 每个关键步骤都应记录清晰日志：业务动作、目标 testid、前置检查结果、后置确认结果、当前状态与状态变化。
- 流程中断时要明确记录失败点、失败原因和上下文，优先报告“前置条件不满足”或“控件不可操作”的具体原因。
- 职责边界保持明确：
  - E2E 负责验证页面元素可达、可操作、可验证。
  - Flow 负责组织可复用业务步骤。
  - CLI 负责提供稳定的外部调用接口。
